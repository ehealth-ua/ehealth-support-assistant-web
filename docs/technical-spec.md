**Проєкт**
- **Назва:**: Інформаційний Web-портал eHealth
- **Джерело вимог:**: `project.prompt` (файл у корені репозиторію)

**Мета**
- **Коротко:**: Створити довідковий портал з навігацією по медичних реєстрах, документації та вбудованою підтримкою ШІ-агентів.
- **Цільова аудиторія:**: медичні адміністратори, розробники реєстрів, користувачі Helpdesk.

**Обсяг (MVP)**
- **Сторінки:**: `Home`, `Registries` (index), `Registry Detail` (для кожного реєстру), `Documentation`, базові `Header`/`Footer`.
- **Компоненти:**: `Header.tsx`, `Footer.tsx`, `Menu.tsx`, `Hero.tsx`, `RegistryCards.tsx`, `DocumentGrid.tsx`, `LanguageSwitcher.tsx`.
- **Мультимовність:**: українська (основна) + англійська, файли перекладів у `i18n/uk.json` та `i18n/en.json`.
- **Інтеграції:**: NotebookLM, Atlassian Helpdesk, Instatus (вбудований iframe).

**Функціональні вимоги**
- **Навігація:**: трирівнева — головне меню → розділ «Реєстри» → сторінка реєстру.
- **Головна сторінка:**: хедер з логотипом «е-Здоров’я», герой-блок, сітка з 7 карток реєстрів, футер з посиланням на Helpdesk.
- **Реєстри (index):**: заголовок «Реєстри: Загальний Каталог», опис, вбудований `iframe` з `https://ekoppho.instatus.com`.
- **Сторінка реєстру:**: назва, герой-блок, дві вертикальні картки: "Аналітичний ШІ (NotebookLM)" та "Підтримка користувачів (Helpdesk)"; кнопки/посилання для переходу до кожної під-функції.
- **Документація:**: список документів з можливістю відкривати у новому вікні (.docx, .pdf, .xlsx, .png тощо).
- **Мова:**: перемикач у хедері, переклади через JSON.

**Нефункціональні вимоги**
- **Адаптивність:**: коректне відображення на mobile/tablet/desktop.
- **Доступність:**: WCAG 2.1 AA (контраст, навігація клавіатурою, ARIA-мітки для основних елементів).
- **Продуктивність:**: LCP < 2.5s для головної сторінки у реальному середовищі; оптимізація зображень, lazy-loading для iframe/документів.
- **Безпека:**: HTTPS, Content Security Policy (CSP) — особливо для `iframe`, валідація й санітизація вхідних даних та параметрів URL, обмеження доступу до сервісних ключів.
- **Розширюваність:**: додавання нових реєстрів через конфіг/контент-адмінку або JSON/YAML файли.

**Архітектура та стек**
- **Frontend:**: `Next.js` + `TypeScript` (рекомендація) або React SPA з SSR/SSG у Next.js для SEO.
- **Styling:**: Tailwind CSS або MUI (залежно від бажаного дизайн-сетапу).
- **i18n:**: `next-i18next` або `react-i18next` з JSON файлами у `i18n/`.
- **Integrations / Backend (рекомендовано):**: невеликий сервер/функції (Node.js/Express або Next.js API Routes) для проксування запитів до NotebookLM та Helpdesk, зберігання секретів у змінних оточення.
- **Хостинг / Деплой:**: Vercel (Next.js) або Azure Static Web Apps / App Service.
- **CI/CD:**: GitHub Actions — build, lint, test, deploy.

**API та модель даних (пропозиція)**
- **Публічні endpoints (при наявності backend):**
  - `GET /api/registries` — список реєстрів (id, name, slug, shortDescription, heroImage)
  - `GET /api/registries/:slug` — деталі реєстру (description, integrations, documents[])
  - `GET /api/documents` — список документів (name, url, type, registryId)
  - `POST /api/helpdesk` — створити тікет (proxied to Atlassian Helpdesk)
  - `POST /api/notebooklm/query` — проксувати запити до NotebookLM (server-side only)
- **Схема реєстру (зразок):**
  - id: string
  - name: string
  - slug: string
  - description: string
  - heroImage: string
  - documents: [{id, name, url, type}]
  - integrations: {notebookLM?: boolean, helpdesk?: boolean, instatus?: boolean}

**UI / Компонування сторінок**
- **`Header.tsx`:** логотип, головне меню, `LanguageSwitcher`, іконка Helpdesk
- **`Footer.tsx`:** контакт, короткий текст «Якщо у користувача виникли запитання…», посилання на політики
- **`Hero.tsx`:** заголовок + підзаголовок + опціональна кнопка CTA
- **`RegistryCards.tsx`:** сітка карток реєстрів (картка: іконка/зображення, назва, короткий опис)
- **`DocumentGrid.tsx`:** сітка документів з іконками типів, відкриття у новій вкладці
- **`Menu.tsx`:** головне меню та опціональна ліва панель (mobile collapse)

**Інтеграції та деталі реалізації**
- **Instatus:** вбудувати `iframe` з `https://ekoppho.instatus.com`. Перевірити CSP та sandbox атрибути (`sandbox="allow-scripts allow-same-origin"`) для безпеки.
- **NotebookLM:** використовувати server-side проксі (щоб прибережно зберігати ключі). UI на сторінці реєстру має кнопку «Запитати аналітику», відкриває modal з input і результатом.
- **Atlassian Helpdesk:** інтегрувати через API для створення/перегляду тикетів або посилатити користувача на зовнішню форму.

**Тестування**
- **Unit:** Jest + React Testing Library для компонентів.
- **E2E:** Playwright або Cypress для перевірки навігації, i18n, відкриття документів, iframe вбудовування.
- **Лінтинг та формат:** ESLint + Prettier + TypeScript checks.

**Деплой і моніторинг**
- **CI:** GitHub Actions — `lint`, `build`, `test`, `deploy`.
- **CD:** автоматичний деплой на Vercel при мерджі в `main` (опціонально з preview-deploy для PR).
- **Моніторинг:** Sentry, логування серверних помилок, uptime checks для зовнішніх інтеграцій.

**Критерії приймання (Acceptance Criteria)**
- **Функціональні:** усі сторінки (Home, Registries, Registry Detail, Documentation) доступні і коректно відображаються; 7 карток на головній; iframe Instatus працює; документи відкриваються у новій вкладці; мова перемикається між українською та англійською.
- **Нефункціональні:** сайт адаптивний, проходить базові тести доступності (ARIA roles, навігація клавіатурою), HTTPS увімкнений, базові тести пройшли у CI.

**Ризики та обмеження**
- **Вбудовані iframe:** можуть бути обмежені політиками сторонніх сервісів (X-Frame-Options).
- **Зовнішні API:** залежність від SLA сторонніх сервісів (NotebookLM, Helpdesk).
- **Дані eHealth:** якщо згодом з'явиться необхідність обробляти чутливі дані, потрібні додаткові вимоги з безпеки та відповідності (рівень сертифікацій).

**Пропозиція етапів і приблизні строки (оціночні)**
- **Sprint 0 (1 тиждень):** погодження ТЗ, скаффолдинг, налаштування i18n, CI.
- **Sprint 1 (1-2 тижні):** реалізація `Home`, `Header`, `Footer`, `RegistryCards`, мобільна адаптація.
- **Sprint 2 (1-2 тижні):** `Registries index`, `Registry Detail` (UI), вбудування iframe, DocumentGrid.
- **Sprint 3 (1 тиждень):** інтеграції (proxy для NotebookLM, Helpdesk), тести, деплой.

**Deliverables (здача)**
- `docs/technical-spec.md` (цей документ)
