# Операційний runbook для супроводу eHealth Portal

## 1. Призначення документа

Цей документ призначений для системних адміністраторів, DevOps-інженерів або технічних супроводжувачів, які мають працювати з репозиторієм та веб-порталом eHealth після передачі проєкту.

## 2. Обсяг продукту

Поточний репозиторій містить веб-портал eHealth, який включає:

- головну сторінку з переліком реєстрів;
- сторінки окремих реєстрів;
- сторінку документації;
- мультимовний інтерфейс;
- посилання на зовнішні сервіси (NotebookLM, Helpdesk, Instatus).

## 3. Важливі розташування

- [web](../web) — основний веб-додаток.
- [web/config/notebooks.json](../web/config/notebooks.json) — конфігурація реєстрів.
- [web/locales/ua.json](../web/locales/ua.json) і [web/locales/en.json](../web/locales/en.json) — локалізації.
- [docs](.) — технічна документація та інструкції.
- [archive/cerebras-legacy](../archive/cerebras-legacy) — застарілі Cerebras-матеріали, які не використовуються у роботі продукту.

## 4. Передумови для запуску

### 4.1. Вимоги

- Node.js 18+ або сумісна версія
- npm
- доступ до мережі для зовнішніх інтеграцій

### 4.2. Локальний запуск

```bash
cd web
npm install
npm run dev
```

### 4.3. Збірка

```bash
cd web
npm run build
```

## 5. Конфігурація та дані

### 5.1. Реєстри

Список реєстрів зберігається у [web/config/notebooks.json](../web/config/notebooks.json).

При зміні списку реєстрів:

1. оновити відповідний запис у JSON;
2. переконатися, що для нового запису є відповідні переклади у [web/locales/ua.json](../web/locales/ua.json) і [web/locales/en.json](../web/locales/en.json);
3. перевірити, чи існує потрібне зображення у папці [web/public/images](../web/public/images).

### 5.2. Локалізація

- основні тексти для української мови — [web/locales/ua.json](../web/locales/ua.json);
- англійська — [web/locales/en.json](../web/locales/en.json).

## 6. Операційні завдання

### 6.1. Перевірка працездатності сайту

Після розгортання або зміни коду варто перевірити:

- відкриття головної сторінки;
- навігацію по реєстрах;
- сторінку документації;
- відображення статус-iframe;
- коректність зовнішніх посилань.

### 6.2. Оновлення вмісту реєстру

Для зміни тексту або посилань:

1. відредагувати [web/config/notebooks.json](../web/config/notebooks.json);
2. при потребі оновити переклади у [web/locales/ua.json](../web/locales/ua.json) та [web/locales/en.json](../web/locales/en.json);
3. перевірити результат через локальний запуск або збірку.

### 6.3. Оновлення документації

Технічні та функціональні документи знаходяться у [docs](.).

Якщо змінюється структура продукту або робочі процеси, оновлювати:

- [docs/technical-spec.md](technical-spec.md);
- [docs/registry-addition-guide.md](registry-addition-guide.md);
- цей runbook.

## 7. Безпека

### 7.1. Що не має потрапляти до репозиторію

Не комітити:

- .env-файли;
- приватні ключі;
- сертифікати;
- токени;
- облікові дані до зовнішніх сервісів.

### 7.2. Рекомендовані практики

- зберігати секрети у змінних середовища або зовнішніх secure stores;
- не зберігати приховані значення у коді;
- перед відправкою змін перевіряти diff на наявність чутливих даних.

### 7.3. Короткий список рекомендацій для безпечної передачі доступу адміністраторам

1. Передавати доступ через індивідуальні облікові записи, а не через спільні паролі.
2. Для доступу до середовища використовувати мінімально необхідний набір прав.
3. Не передавати секрети у чатах, email або текстових файлах без захисту.
4. Після передачі доступу перевірити, чи не залишилися в репозиторії локальні .env-файли, ключі або токени.
5. Якщо доступ передається стороннім адміністраторам, зафіксувати дату, особу, рівень доступу та дату відключення/перегляду.
6. Передавати лише той доступ, який потрібен для супроводу, а не повний доступ до всіх середовищ без потреби.

## 8. Розгортання

Після збірки та перевірки:

1. забезпечити доступ до потрібного середовища;
2. розгорнути актуальний build;
3. перевірити доступність головної сторінки;
4. перевірити, що посилання на зовнішні сервіси працюють.

## 9. Типові проблеми

### 9.1. Сторінка не відкривається

- перевірити, що збірка пройшла;
- перевірити журнали застосунку;
- перевірити, чи не зламана конфігурація маршрутизації.

### 9.2. Не відображається реєстр

- перевірити запис у [web/config/notebooks.json](../web/config/notebooks.json);
- переконатися, що є відповідні переклади;
- перевірити, чи є зображення для картки.

### 9.3. Порушено роботу зовнішніх посилань

- перевірити доступність зовнішнього ресурсу;
- перевірити, чи посилання не вказують на застарілий URL;
- при потребі оновити конфігурацію.

## 10. Контрольні точки після змін

Після внесення змін у код або контент обов’язково:

- виконати збірку;
- перевірити головну сторінку;
- перевірити сторінку реєстру;
- перевірити сторінку документації;
- перевірити наявність чутливих даних у diff.
