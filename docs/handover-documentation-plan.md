Дата: 2026-06-23
Статус: чернетка для ознайомлення та погодження

## 1. Короткий висновок

Після огляду репозиторію видно, що активний продукт у цьому сховищі вже не є чистим розширенням Cerebras, а є веб-порталом eHealth, реалізованим як Next.js-додаток у папці [web](../web).

Натомість ключові файли у корені репозиторію досі описують початковий проєкт на базі Cerebras, що створює плутанину для майбутнього супроводу.

## 3. Огляд безпеки

### 3.1. Що я не знайшов

Під час перегляду репозиторію не було виявлено очевидних файлів із реальними секретами у форматі:

- .env
- .pem
- .key
- .p12
- .pfx

Також не було виявлено очевидних закомічених токенів або паролів у основному коді проєкту.

### 3.2. Що потребує уваги

Серед ризиків є наступне:

1. В кореневому [ .gitignore ](../.gitignore) відсутні правила для приховування файлів типу .env, .env.*, *.pem, *.key, *.p12.
2. У папці [docs/raw](raw) є приклади з називами на кшталт secret / token / API key; їх варто перевірити, щоб не залишити у репозиторії матеріал, який може сплутати або створити зайвий ризик для передачі.
3. У коді є логіка роботи з API-ключем через VS Code Secrets API в [src/provider.ts](../src/provider.ts); це безпечно для локального запуску, але не має відношення до веб-порталу і не повинно бути головним контекстом для адміністраторів.

### 3.3. Рекомендація по безпеці перед передачею

До того, як передавати репозиторій, варто:

- додати правила у .gitignore для локальних секретів;
- переконатися, що жодні локальні .env-файли не лежать у репозиторії;
- якщо коли-небудь секрети вже були закомічені в історію, провести їх видалення з історії Git перед публікацією або передачею доступу.

## 4. Що варто прибрати, а що залишити

### 4.1. Файли, які рекомендую замінити або перезаписати

- [README.md](../README.md) — замінити на новий кореневий README про eHealth-портал, його архітектуру, запуск, деплой і точки підтримки.
- [CONTRIBUTING.md](../CONTRIBUTING.md) — замінити на інструкцію для розробників/адмінів, які працюють із веб-порталом.
- [SUPPORT.md](../SUPPORT.md) — або замінити на короткий документ підтримки для системних адміністраторів, або прибрати з основного шляху handover.

### 4.2. Файли, які варто зберегти, але адаптувати

- [docs/technical-spec.md](technical-spec.md) — зберегти як базовий технічний опис продукту.
- [docs/registry-addition-guide.md](registry-addition-guide.md) — зберегти як operational guide для розширення реєстрів.
- [web/README.md](../web/README.md) — залишити, але оновити, щоб він був ближчим до реальної експлуатаційної інструкції.

### 4.3. Файли, які варто розглядати як legacy/архів

Якщо у майбутньому планується чистий handover-репозиторій, варто або:

- перенести старі Cerebras-артефакти в окрему папку archive/legacy, або
- прибрати їх з активного шляху репозиторію, якщо вони не потрібні для супроводу.

## 5. Рекомендований план дій

### Етап 1. Визначення меж продукту

- підтвердити, що активний продукт — це веб-портал eHealth;
- рішення: або залишити в репозиторії обидва напрямки (legacy + active), або відокремити legacy у архів.

### Етап 2. Очищення документації

- створити новий кореневий README для продукту;
- замінити або видалити застарілі Cerebras-сторінки;
- залишити тільки ту документацію, яка корисна для розробників, адміністраторів і супроводу.

### Етап 3. Безпекова підготовка

- додати правила у .gitignore для .env і секретних файлів;
- перевірити, що в репозиторії не залишилося локальних секретів;
- якщо потрібно — очистити історію Git від конфіденційних даних.

### Етап 4. Підготовка handover-пакету для адміністраторів

До складу handover-пакету рекомендую включити:

1. README про продукт і його призначення.
2. Короткий розділ про архітектуру і залежності.
3. Інструкцію з запуску, збірки і деплою.
4. Список необхідних змінних середовища та місце зберігання секретів.
5. Короткий runbook для типових операційних завдань.

## 7. Пропозиція щодо порядку узгодження

Рекомендую узгодити в такому порядку:

1. чи залишати legacy Cerebras-частину в активному репозиторії, чи перенести її в архів;
2. який саме текст має бути у новому кореневому README;
3. чи потрібен окремий адміністративний runbook, або достатньо буде короткого розділу в README;
4. що саме вважати критичним для безпечної передачі доступу системним адміністраторам.

- список змін до .gitignore і рекомендації щодо безпечного супроводу.
